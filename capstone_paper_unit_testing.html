Overall System Unit Testing and Quality Assurance

1. Introduction to the Comprehensive Testing Strategy

The Shoelotskey Service Management System strictly employs an extensive Unit Testing methodology designed to guarantee zero data-loss, correct financial mathematics, and consistent system stability across all operational modules. Because the system utilizes a high-availability synchronized database layer (AWS PostgreSQL online shifting to local SQLite offline), rigorous overall testing of all endpoints and UI components is highly critical.

Tests isolate all components. Frontend user-interface tests are evaluated over virtual DOM instances (via React Testing Library), while backend operations are verified strictly using Python's PyTest library, simulating payload manipulation over isolated, transient testing databases.

2. Module-by-Module Testing Execution Plan

Below is the detailed architectural breakdown outlining precisely how each core module of the system is independently verified and rigorously tested prior to deployment.

A. Authentication & User Management Module

Ensures that sensitive system access is heavily fortified. These tests focus on cryptographic security and role-access control.

  • Credential Validation: Test that injecting incorrect passwords or invalid JWT tokens results in an immediate HTTP 401 Unauthorized block.
  • Role-Based Access Control (RBAC): Ensure that endpoints designated exclusively for an `Owner` role successfully reject requests executed by accounts strictly classified as `Employee`.

B. Inventory Management Module

Ensures robust supply chain oversight by verifying that stock levels perfectly correspond to real-life tracking and database insertion rules.

  • Item Creation Check: Verify that new inventory items pass strict string-length checks and reject negative numerical constraints in the 'quantity' parameter.
  • Job Order Stock Deduction Trap: Programmatically simulate a Job Order confirming the consumption of 1 unit of 'Sole Glue', then immediately assert that the backend SQL implicitly modifies the Inventory table by `-1` unit accurately.

C. Service Layout & Pricing Module

Ensures that the catalog of available shoecare services maintains structural integrity across the system.

  • Mathematical Dependency Test: Submits a test service with Add-ons (e.g. Deep Cleaning Base Cost + Laces Restoring Add-On) and ensures the FastAPI sum calculations equate precisely before returning the overall Grand Total to the frontend payload.
  • Status Toggle Reliability: Asserts that soft-deleted or locally archived services successfully cease populating the active frontend Dropdown Selection UI during a new Job Order.

D. Expense Tracking Module

Guarantees that operational business expenses are accurately recorded, properly bound to specific dates, and completely tamper-resistant against incorrect financial logging.

  • Categorization Validation: Test that an expense logs successfully only if it matches pre-defined Enums (e.g., Marketing, Supplies, Utility) and forcefully blocks invalid category strings.
  • Date Back-Log Audit: Verify that admins can safely log historical expenses prior to today's current date without crashing SQLite internal calendar structures.

E. Core Job Order & Machine Learning (ML) Module

This is the central nervous system. Tests must extensively validate form resilience, payment verification, and the Machine Learning algorithmic decision math.

  • Priority Queue Math Test (ML Focus): Simulate sending a Job Order tagged as 'Rush Priority'. Assert that the ML predictive function definitively subtracts queue days (e.g., generating a mathematically shorter duration than 'Regular Priority').
  • Payment Mathematical Integrity: Provide an order totaling ₱3000. Forcefully select 'Downpayment (50%)' and evaluate that the React Frontend forces the exact integer `1500` strictly into the read-only Amount constraints.
  • Receipt Token Injection: Verify identical data replication upon generating Customer tracking tokens.

F. Dashboard Analytics & Sales Report Module

Ensures highly graphical analytics generate from perfectly computed backend analytical data.

  • Stored Procedure Reliability: Fire the PostgreSQL Stored Procedure generate_daily_sales_summary entirely stripped of real traffic, inputting mock sales data. Assert that it effectively clumps identical calendar dates into cohesive single-row revenue aggregations.
  • Live Data Rendering: Assures that any asynchronous fetch resolving to 'Null' beautifully defaults React charts to "Zero value / No data" fallbacks instead of causing application white-screens.
